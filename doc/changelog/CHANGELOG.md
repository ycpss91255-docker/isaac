# Changelog

## [Unreleased]

### Removed
- **Same-repo multi-instance dropped entirely (ADR-0019, supersedes ADR-0016's multi-instance premise).** isaac is single-scenario validation, so the `--instance <name>` flow is removed: the committed `config/instances/example.{yaml,env}` templates are deleted, `script/hooks/pre/run.sh` reverts to a no-op stub (its only job was the per-instance cache-dir tree), and `script/hooks/{post/run,post/stop}.sh` are de-instanced to the default-only path -- Isaac container `${USER_NAME}-${IMAGE_NAME}-stream`, viewer container `owv`, viewer ports literal `-e SIGNALING_PORT=49100 -e SERVE_PORT=5173` (the former fallback), `VIEWER_UI_MODE=stream-only` + image `${DOCKER_HUB_USER:-local}/omniverse_web_viewer:runtime` unchanged. The supported flow is now `just run -t stream -d` (no `--instance`) + an explicit `exec` to launch Isaac, then `just stop`. The only real driver for same-repo multi-instance was test-time port-collision avoidance, not a product need (multi-robot -> Isaac Lab RL multi-env in one process per ADR-0018; multi-user -> copy the repo; scenarios run one at a time). base keeps the `--instance` primitive (base #465; de-leaking it from base is tracked in base #600) and the full multi-instance design is preserved for future non-isaac repos in multi_run#15. ADR-0016's base-native wrapper-hook adoption (base #440) otherwise stands. Specs: `test/smoke/bats/pre_run_hook_spec.bats` deleted (no-op stub); `post_run_hook_spec.bats` (12 -> 10) and `post_stop_hook_spec.bats` (2 -> 1) rewritten to default-only; the `pre/run.sh` Dockerfile devel-test COPY dropped. bats smoke total 64 -> 57.

### Added

(Pinned to `v2.3.0`, not the latest 2.3.x: Isaac Lab PR #4000 -- first shipped in `v2.3.1` -- made `UrdfConverter` hard-enable the URDF importer extension `isaacsim.asset.importer.urdf-2.4.31`, but `isaac-sim:5.1.0` bundles `2.4.30` and the offline GPU runner cannot fetch `2.4.31` from the Kit registry, so `UrdfConverter` raises `AttributeError: set_merge_fixed_ignore_inertia`; `v2.3.0` predates #4000 and drives the bundled `2.4.30` importer. A GPU integration smoke (`test/integration/pytest/test_isaaclab_available.py`) launches `AppLauncher` headless and asserts `isaaclab.sim` imports with the spawner + `UrdfConverterCfg` surfaces and the `2.3.x` version (GPU baseline 26 -> 27). The framework (`isaac_devkit`) stays mounted, not baked (ADR-0017 section 2: base tools baked + pinned, framework mounted). This is the foundation for the MR re-base (#150-#154).

### Changed
- **`build_scene` is rewritten as a YAML -> Isaac Lab `sim_utils` cfg adapter; the raw-`pxr` spawn path is removed (#152, ADR-0018 decisions 1 & 3).** `isaac_devkit.scene` gains a new PURE adapter `to_isaaclab_cfg(scene, repo_root) -> list[SpawnSpec]` that maps the validated three-file scene dict (schema UNCHANGED) to a neutral, JSON-serializable `SpawnSpec` NamedTuple list (`prim_path`, `kind` in `{ground_plane, distant_light, usd}`, `kwargs`, `translation`, `orientation` `(w,x,y,z)`, `mobility`). It emits, in order: the environment ground plane (`/World/ground`, when `environment.ground_plane`), a distant light (`/World/light`, from `environment.light`'s curated `intensity`/`color`/`angle`, or a default `intensity=3000` SunLight when absent), the robot USD (`/World/Robot`), then one USD spec per object instance (`/World/Objects/<modelstem>_<idx>_<instidx>`, carrying `mobility`). Curated fields map to cfg kwargs (`model` -> `usd_path`, `pose.xyz` -> `translation`, `pose.rpy` deg -> `orientation` via the new pure `rpy_to_quat` XYZ-intrinsic helper matching the old `AddRotateXYZOp`); `spawn_overrides:` (a dict on any entry) is spread onto kwargs LAST so raw passthrough wins over curated fields (intentional power-user control); a referenced material YAML's diffuse color (via #153's `material_cfg_from_yaml`) is recorded under `visual_material_diffuse` when present, else skipped. The adapter imports NO Isaac module -- it is the stable seam (ADR-0018: the spec list is C-upgradeable into an `InteractiveSceneCfg` without changing the schema or contract), asserted by a hosted unit test without a live stage. `build_scene(scene, stage, repo_root)` (signature UNCHANGED) now calls the adapter and materializes each `SpawnSpec` into the matching `isaaclab.sim` cfg (`DistantLightCfg` for light, `UsdFileCfg` for robot/object) with `cfg.func(prim_path, cfg, translation=, orientation=)`; the `ground_plane` spec spawns a PROCEDURAL collision cuboid floor (`CuboidCfg`), NOT Isaac Lab's `GroundPlaneCfg` -- whose default `usd_path` is a nucleus asset (`ISAAC_NUCLEUS_DIR/.../default_environment.usd`) that the offline base-repo GPU runner cannot fetch (it resolves to None and the spawn dies with `ArgumentError GetPrimAtPath(Stage, NoneType)`); a caller with a real ground USD can pass `usd_path` via `environment.ground_overrides` (-> `UsdFileCfg`); `mobility: dynamic` attaches `rigid_props` + `mass_props` + `collision_props`, `static` attaches `collision_props` only, `None` attaches none; `visual_material_diffuse` becomes a `PreviewSurfaceCfg(diffuse_color=...)`. The raw-`pxr` `DefinePrim` / `GetReferences().AddReference` / `_apply_pose` / variant-selection path is removed (`variant:` now flows through as a `UsdFileCfg` `variants` kwarg for structurally-distinct variants); all `isaaclab.sim` imports stay function-local. The driver's `_ensure_scene_defaults` SunLight default is made opt-out (skips when a light already exists at `/World/light` or `/World/SunLight`) so the adapter light does not double up; the example-driver consolidation is #154. The import-safety invariant is tightened to also ban module-top `isaaclab` imports (runtime + AST guards + ruff TID253). `resolve_model_path` is made tolerant of a value that already carries the `model/usd/` segment (the example scene's robot entry uses `model/usd/robot/camera_bot/camera_bot.usd`, its object entry the bare `object/prop_cube/prop_cube.usda` -- both resolve under `<repo_root>/model/usd/`); #154 normalizes the example scene to the bare convention when it rewrites the example driver. Tests: `test_scene_builder.py` adds `TestToIsaaclabCfg` + `TestRpyToQuat` + the `resolve_model_path` shim test (17 -> 39); hosted-unit baseline ratchets 233 -> 255. The adapter gets dedicated GPU coverage in `test/integration/pytest/test_build_scene_adapter.py` (3 tests via `_build_scene_runner.py`): the example scene's environment ground (a procedural collision cuboid) + light + robot USD spawn on a live stage through `to_isaaclab_cfg` -> `cfg.func()`, and a `mobility: dynamic` object materializes its `rigid_props`/`mass_props`/`collision_props` cfg and spawns -- exercising the live `cfg.func` spawn independently of the example driver's own raw-pxr `_build_scene` (GPU-integration baseline 25 -> 28). The dynamic object's `RigidBodyAPI` and the robot's `base_link` are reported, not asserted: the committed example USDs are legacy-importer assets with no `defaultPrim`, so the `UsdFileCfg` references resolve empty until #154 sets their `defaultPrim` (and, since Isaac Lab's `UsdFileCfg` rigid-props only `modify_*` an existing API, defines the mobility physics post-spawn). Reasoning: (a) `headless` can't host this — its ENTRYPOINT already runs `runheadless.sh -v` so a second kit from `python.sh` would collide on PhysX / DDS / port 8211; (b) `devel` can't host this either — base v1's `[stage:devel]` is reserved and rejects per-stage overrides ("[setup] WARNING: [stage:devel] is reserved; not applied in v1") so we can't strip X11 from devel without breaking the rest of the devel workflow.
- `setup.conf [stage:standalone] gui.mode = off` — per-stage override mirrors `[stage:headless]`'s pattern, strips X11 mount + `DISPLAY` env from the standalone service so kit's `carb.windowing-glfw` / X11-reject cosmetic warnings stay off during livestream startup. The `gui` stage (which legitimately needs X11 for `runapp.sh`) keeps the default `[gui] mode = auto`.
- `setup.conf [environment]` ROS 2 bridge baseline (distro-agnostic) — `ROS_DOMAIN_ID=0`, `RMW_IMPLEMENTATION=rmw_fastrtps_cpp`, `FASTRTPS_DEFAULT_PROFILES_FILE=/isaac-sim/fastdds.xml`. Distro choice itself moved to `setup.conf [build] arg_N=ROS_DISTRO=...` (build-time, hard-baked — see `Distro choice` Notes entry). Aligns with NVIDIA's official [Isaac Sim 5.1 ROS 2 install guide](https://docs.isaacsim.omniverse.nvidia.com/5.1.0/installation/install_ros.html).
- `setup.conf [build] arg_4=ROS_DISTRO=humble` — distro is a build-time choice. `ARG ROS_DISTRO` in Dockerfile defaults to humble; switching = edit `setup.conf` + `./build.sh` (rebuilds only the affected layers, ~10s).

Reads `/etc/isaac/ros-distro` (baked at build time from `ARG ROS_DISTRO`) and unconditionally exports `ROS_DISTRO` + `LD_LIBRARY_PATH=/isaac-sim/exts/isaacsim.ros2.bridge/${ROS_DISTRO}/lib` before exec'ing the wrapped binary. ### Fixed
- Dockerfile `standalone` stage: add explicit `CMD ["sleep", "infinity"]`. Without this the standalone profile-gated compose service inherited devel's `bash` CMD but with `stdin_open: false, tty: false` (base #215 profile-gated service default, vs the base devel service's `true/true`), so `bash` ran non-interactively and exited 0 immediately on `./run.sh -t standalone -d`. Symptom: `docker ps -a` showed `<user>-isaac-standalone Exited (0) <N> seconds ago`, then `./exec.sh -t standalone <cmd>` aborted with `Container '<user>-isaac-standalone' is not running` — even though `./run.sh` reported `Container <user>-isaac-standalone Started` (compose's `Started` only acknowledges the start request, not container-stays-up). `ENTRYPOINT` stays inherited from devel (`/entrypoint.sh`) — entrypoint does env init then `exec "$@"`, which now resolves to `sleep infinity` and keeps the container idle until `./stop.sh` reclaims it. Headless / gui escape this trap because they explicitly set `runheadless.sh -v` / `runapp.sh` as long-lived ENTRYPOINT/CMD.

### Notes
- Isaac Sim 5.1 reads privacy consent from `OMNI_ENV_PRIVACY_CONSENT`, not `PRIVACY_CONSENT`. The latter is preserved for backwards compat but no longer effective; set the former explicitly to opt in to telemetry.
- `[stage:headless] gui.mode = off` requires template **v0.18.1 or later** — v0.18.0 had a list-inheritance bug where the override was emitted on top of `extends: devel`, so X11 mounts inherited regardless. Fixed in v0.18.1 by switching to standalone-emit when any list-affecting override fires.
- ROS 2 bridge auto-loads via the default kit experience (`isaacsim.exp.full.kit` → `[settings] isaac.startup.ros_bridge_extension = "isaacsim.ros2.bridge"`); no `--enable isaacsim.ros2.bridge` launch flag is needed in `runheadless.sh` / `runapp.sh` CMD.
- Distro choice (build-time, hard-baked): default `humble` via `setup.conf [build] arg_4=ROS_DISTRO=humble`. Switch to jazzy = edit `setup.conf` + `./build.sh` rebuild (only the ENV / RUN-echo layers rebuild, not the 15 GB Isaac Sim base — ~10s). `headless` / `gui` ENTRYPOINT goes through `isaac-ros-env-wrapper.sh` which re-exports from `/etc/isaac/ros-distro` at every container start, so runtime `-e ROS_DISTRO=...` flags do not take effect. Bridge distro priority (per `isaacsim.ros2.bridge/extension.py`): `os.environ["ROS_DISTRO"]` > kit setting `exts."isaacsim.ros2.bridge".ros_distro` (resolves `system_default` → Ubuntu detection); the wrapper guarantees the env-var path wins with the baked value.
- Why a wrapper shim instead of `Dockerfile ENV` alone — Docker's `ENV` is overridable at runtime (`docker run -e ...` / compose `environment:`), which would silently mismatch with the baked `LD_LIBRARY_PATH` if a downstream user passed `-e ROS_DISTRO=jazzy`. The wrapper short-circuits that footgun by always re-reading from the build-time const file.
- Template upgraded v0.18.1 → v0.20.1: ships the `${KEY}` cross-reference fix above, plus `setup_tui` main-menu restructure (cosmetic) and an opt-in `publish-worker.yaml` reusable workflow (we don't consume it).
