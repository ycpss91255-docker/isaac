# Changelog

## [Unreleased]

### Changed
- **BREAKING: base subtree upgraded `v0.40.0` -> `v0.41.0`; wrapper entrypoint moves from `make` to `just`.** The root `Makefile` symlink is retired and replaced by a `justfile` symlink (base #475, ADR-0005). Use `just build|run|exec|stop|setup|upgrade` (run `just` for the list). The legacy `runtime.env` file is retired: `setup.sh` now generates `.env` (workload overlay) + `.env.generated` (derived cache) instead (base #497/#507 S2/S7); both are gitignored derived artifacts. The `[deploy] runtime` setup.conf key is renamed to `gpu_runtime` (old key kept as a permanent alias, removal targeted at base v1.0.0). Verified clean: a fresh `init.sh` from v0.41.0 and this in-place upgrade produce byte-identical `.base/` trees and matching base-managed artifacts (symlink set, `.env.generated` keys modulo the repo's own `ROS_DISTRO` override, no `runtime.env`). Unblocks the GPU `run.sh -t test` pytest mechanism (base #493) for downstream consumers.
- **BREAKING: multi-instance bring-up moves to base-native `run.sh --instance` + wrapper hooks** (adopts base #465 + #440; see ADR-0016). The hand-rolled `init_instance.sh` / `run_instance.sh` / `stop_instance.sh` and the `Makefile.local` `run-stream` / `stop-stream` targets are retired. New flow: author `config/instances/<name>.{yaml,env}` from the committed `example.{yaml,env}` template, then `./run.sh -t stream -d --instance <name>`. The per-instance cache tree is created by `script/hooks/pre/run.sh`; `host.yaml` injection + web-viewer launch by `script/hooks/post/run.sh`; web-viewer teardown by `script/hooks/post/stop.sh` (stopped via `./stop.sh --instance <name>`). Isaac Sim launch stays an explicit `exec` step (driver per ADR-0005, or `runheadless-host-config.sh` for the no-driver case) -- `run` brings up infra, `exec` runs the workload. Migrate existing `config/docker/instances/<name>.env` to `config/instances/<name>.{yaml,env}` (one-time, manual).
- `runheadless-host-config.sh` is now the single builder of the livestream Kit args: it reads `ISAAC_SIGNAL_PORT` / `ISAAC_MEDIA_PORT` / `ISAAC_API_PORT` from the container env (delivered by the per-instance compose overlay) and `public_ip` from `/etc/host.yaml`, emitting the full `--/app/livestream/*` invocation. The duplicated arg construction in the retired `run_instance.sh` is gone.
- Bundled `web_viewer` submodule bumped `a649dc8` -> `f371ee5` (omniverse_web_viewer #17 + #15 + #24 + #25). The viewer entrypoint now validates every config value and fails fast on bad input, strips inline `# comments` from `host.yaml`, and re-renders the served bundle from pristine templates on every boot (idempotent -- a `docker restart` or changed env / host.yaml is picked up next start, instead of the old one-shot in-place edit). An embeddable stream demo (`examples/embedded-site-demo`, served on 8080, with a 4-language README) is also now available in the submodule. Backward-compatible for `run-stream` / `run_instance.sh` (precedence host.yaml > env > default unchanged); a fresh `git submodule update --init` now checks out this entrypoint.
- Scrubbed a real LAN IP that had leaked into illustrative content: the host.yaml example comment now uses a generic `192.168.50.1`, and the `host_yaml_spec.bats` / `isaac_ros_env_wrapper.bats` fixtures use `127.0.0.1`. No real address ships in the repo's docs/tests. (Functional configs that genuinely need a host address -- `config/ros2/fastdds_tcp_client.xml`, `script/view_isaac_camera.sh`, ADR-0015 -- are left for a separate parameterization pass.)

### Fixed
- Dockerfile devel stage now bakes `~/.cache` and `~/.nv` user-owned into the image (found by #127's integration test). The compose volumes mount subpaths (`~/.cache/ov`, `~/.cache/pip`, `~/.cache/nvidia/GLCache`, `~/.nv/ComputeCache`); with the parent dir absent from the image, dockerd mkdirs it as root at container start, and anything else writing under `~/.cache` then fails. Concretely: warp's `~/.cache/warp` PermissionError killed `omni.replicator.core` at extension startup, which unregistered the SDG node templates (`DispatchSync` etc.) that the ROS2 camera publish chain needs -- the OmniGraph built fine but zero frames were ever published (writer attach failed with "graph node template DispatchSync unregistered").
- `script/run_instance.sh`: two compounding bugs that produced a working-looking output ("Container started", "Web-viewer started") while the Isaac container had already died with `exitCode=2 / execDuration=0` and the browser showed a white screen (closes #81).
  - **Bug A**: `kit_args` array used to start with `-v`. The image entrypoint is `exec "$@"`, so the bash exec builtin treated `-v` as its own verbose flag and the container died before Isaac was invoked. Fix: prepend `/isaac-sim/runheadless.sh` to `kit_args`, mirroring what `Makefile.local`'s `run-stream` exec does from the other direction.
  - **Bug B**: `_start_web_viewer` did not pass `-e SIGNALING_SERVER=${public_ip}` to the viewer container. With a stale locally cached `owv:runtime` image (built before `omniverse_web_viewer#12`, the entrypoint that reads `/etc/host.yaml`), the viewer JS bundle silently fell back to `SIGNALING_SERVER=127.0.0.1`. Fix: pass `SIGNALING_SERVER` as defense in depth; the post-`#12` entrypoint still prefers `/etc/host.yaml`, so this is a backward-compatible fallback.
  - Regression guarded by `test/smoke/bats/run_instance_spec.bats`, with `script/run_instance.sh` baked into `/smoke_test/` at devel-test build time (same pattern as `makefile_local_spec.bats` under #75).
  - Stale viewer image auto-rebuild is tracked as a separate follow-up; current docs note that `owv:runtime` must be rebuilt by hand after the `web_viewer/` submodule pointer bumps.
- `Makefile.local run-stream`: docker-logs FD redirect now targets the container PID 1's host-side PID resolved via `docker inspect --format '{{.State.Pid}}'`, instead of the literal `/proc/1/fd/{1,2}`. The container runs `pid: ${PID_MODE}` with `PID_MODE=host`, so `/proc/1/...` from inside the container resolves to host systemd; writes get rejected with EPERM and `docker logs <isaac>` / lazydocker stayed empty while Isaac Sim was actually running. With the corrected target, Kit output streams into `docker logs` natively (closes #75). Regression guarded by `test/smoke/bats/makefile_local_spec.bats`, with `Makefile.local` baked into `/smoke_test/` at devel-test build time.
- `host.yaml` `public_ip` parsing hardened (closes #104). The inline awk parser (duplicated in `run_instance.sh` and `runheadless-host-config.sh`) did not strip a trailing inline `# comment`, did not trim, and did not validate, so a realistic config -- the template itself encourages inline comments -- produced a garbage value (`1.2.3.4  # note`) that Kit and the web-viewer silently rejected, breaking WebRTC with no diagnostic. Both call sites now source one shared `script/host_yaml.sh` (`resolve_public_ip`): strips the comment + whitespace + quotes, charset-validates (errors out on garbage instead of passing it on), and warns when the key is present but unparseable (vs silently treating it as "not configured"). Regression guarded by `test/smoke/bats/host_yaml_spec.bats`.
- Web-viewer launch in `run_instance.sh` made idempotent and stage-gated (closes #105). `_start_web_viewer` now `docker rm -f`s a stale viewer container before `docker run` (mirroring `Makefile.local run-stream`), so a re-run no longer fails with a name conflict; and the launch is gated on the `stream` stage, so `headless` (no stream) no longer spawns a viewer. Regression guarded by `run_instance_spec.bats`.
- `script/hooks/post/run.sh` (base #440 post-run hook): web-viewer launch used the renamed/stale image `owv:runtime`; a host left with that old image (built before the host.yaml-reading entrypoint) makes a remote browser show a white screen because the rendered bundle falls back to `SIGNALING_SERVER=127.0.0.1`. The hook now launches `${DOCKER_HUB_USER:-local}/omniverse_web_viewer:serve` (matching compose image naming); a fresh `serve` image reads `/etc/host.yaml` `network.public_ip` into the bundle. Closes #121. Regression guarded by `test/smoke/bats/post_run_hook_spec.bats`. (Note: this hook does not yet fire for `run.sh -t stream -d` -- tracked as `ycpss91255-docker/base#537`.)

### Notes
- Isaac Sim 5.1 reads privacy consent from `OMNI_ENV_PRIVACY_CONSENT`, not `PRIVACY_CONSENT`. The latter is preserved for backwards compat but no longer effective; set the former explicitly to opt in to telemetry.
- `[stage:headless] gui.mode = off` requires template **v0.18.1 or later** — v0.18.0 had a list-inheritance bug where the override was emitted on top of `extends: devel`, so X11 mounts inherited regardless. Fixed in v0.18.1 by switching to standalone-emit when any list-affecting override fires.
- ROS 2 bridge auto-loads via the default kit experience (`isaacsim.exp.full.kit` → `[settings] isaac.startup.ros_bridge_extension = "isaacsim.ros2.bridge"`); no `--enable isaacsim.ros2.bridge` launch flag is needed in `runheadless.sh` / `runapp.sh` CMD.
- Distro choice (build-time, hard-baked): default `humble` via `setup.conf [build] arg_4=ROS_DISTRO=humble`. Switch to jazzy = edit `setup.conf` + `./build.sh` rebuild (only the ENV / RUN-echo layers rebuild, not the 15 GB Isaac Sim base — ~10s). `headless` / `gui` ENTRYPOINT goes through `isaac-ros-env-wrapper.sh` which re-exports from `/etc/isaac/ros-distro` at every container start, so runtime `-e ROS_DISTRO=...` flags do not take effect. Bridge distro priority (per `isaacsim.ros2.bridge/extension.py`): `os.environ["ROS_DISTRO"]` > kit setting `exts."isaacsim.ros2.bridge".ros_distro` (resolves `system_default` → Ubuntu detection); the wrapper guarantees the env-var path wins with the baked value.
- Why a wrapper shim instead of `Dockerfile ENV` alone — Docker's `ENV` is overridable at runtime (`docker run -e ...` / compose `environment:`), which would silently mismatch with the baked `LD_LIBRARY_PATH` if a downstream user passed `-e ROS_DISTRO=jazzy`. The wrapper short-circuits that footgun by always re-reading from the build-time const file.
- Template upgraded v0.18.1 → v0.20.1: ships the `${KEY}` cross-reference fix above, plus `setup_tui` main-menu restructure (cosmetic) and an opt-in `publish-worker.yaml` reusable workflow (we don't consume it).
